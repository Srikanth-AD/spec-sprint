## Summary

This project adds Google and Microsoft OAuth SSO (via PKCE Authorization Code flow) to an existing email/password SaaS application. It includes account linking for existing password users, org-level admin controls to enable/force SSO, and a comprehensive auth event audit log. SAML, Okta, SCIM, and custom IdP support are explicitly out of scope.

## Assumptions

- The app already has a concept of organizations (orgs) with admin roles; we are extending existing org infrastructure, not building it.
- The team's existing JWT issuance and refresh token logic is encapsulated in a reusable service that can be called from new OAuth callbacks.
- A server-side session store (e.g., Redis or DB-backed) is available to hold short-lived PKCE verifiers and state tokens; we are not relying solely on cookies for this.
- The frontend has an existing auth interceptor that handles 401 responses; we are extending it, not replacing it.
- OAuth client credentials (client_id, client_secret) for Google and Microsoft will be provisioned by the team prior to development; this ticket set does not cover IdP app registration.
- The E2E testing framework supports intercepting and mocking HTTP requests to external IdPs (e.g., via MSW or equivalent).
- HTTPS is already enforced in all environments; no tickets are needed for TLS configuration.
- Unlinking an SSO provider from an account is explicitly out of scope for this sprint.
- The 'common' Microsoft tenant is sufficient; per-tenant enterprise app registration is out of scope.
- Existing TOTP MFA is unchanged and will continue to apply after SSO sign-in where it was previously configured for a user.

## Risks

- Google and Microsoft JWKS endpoints can have key rotation events; if the JWKS cache is not implemented with proper TTL and forced-refresh on unknown kid, token validation will start failing in production.
- Force-SSO is a high-impact feature: if enabled incorrectly or if an org admin enables it before SSO is fully configured, members can be locked out. A warning and dry-run check should be considered.
- Microsoft's 'common' tenant endpoint validates issuer loosely; if iss validation is not tightened appropriately for the token version (v1 vs v2), it could allow tokens from unintended tenants.
- The account-linking flow relies on a 15-minute linking token; if the user flow is interrupted (e.g., browser closed), they will need to restart the SSO sign-in, which may be confusing without clear UX guidance.
- Sprint velocity of 32 points is tight for this feature set. Total estimated points are approximately 79, requiring roughly 2.5 sprints. Prioritization should ensure EP-1, EP-2, and core EP-3 land in Sprint 1.
- Live integration testing with real Google and Microsoft dev apps adds coordination overhead; if OAuth app provisioning is delayed, SS-4 through SS-8 are blocked.
- Audit log writes that fail silently (fire-and-forget) could create compliance gaps; the team should decide whether any event types require synchronous write guarantees.

#### `SS-3` — PKCE utility: code verifier and challenge generation

**2 pts** · backend · high · Sprint 1

> As a backend engineer, I want a shared PKCE utility module so that both the Google and Microsoft OAuth flows can generate and validate code verifiers and challenges consistently.

Implement a utility module that generates a cryptographically random code verifier (43–128 chars, URL-safe base64), derives the S256 code challenge, and stores the verifier in the server-side session for later exchange. This module should be fully unit-tested and independent of any specific IdP.

**Acceptance criteria:**

- Given The PKCE utility is called, when A code verifier is generated, then It is between 43 and 128 characters, URL-safe base64 encoded, and cryptographically random (verified by entropy check in tests)
- Given A code verifier is provided, when The challenge is derived, then The output matches SHA-256(verifier) base64url-encoded per RFC 7636
- Given The unit test suite runs, when All PKCE utility tests execute, then 100% of tests pass with no test relying on fixed/seeded random values

_Estimate rationale: Small, self-contained crypto utility with thorough unit tests; approximately one day of work._

#### `SS-8` — User lookup, auto-provisioning, and JWT session issuance post-SSO

**5 pts** · backend · high · Sprint 2

> As a user who signs in via SSO for the first time, I want an account to be automatically created for me so that I can access the app without a separate registration step.

After successful token validation (SS-5 or SS-7), implement shared service logic to: look up the user by verified email, create a new user record if none exists, create an `sso_identities` row for the provider/subject, then issue the existing JWT session token. Session expiry should be the minimum of 24 hours and the ID token's exp. This logic should be shared by both provider callbacks.

**Acceptance criteria:**

- Given An SSO callback completes with a verified email that does not exist in the system, when User provisioning runs, then A new user record and sso_identities row are created and a valid JWT is issued
- Given An SSO callback completes with an email that already has a linked SSO identity for that provider, when Lookup runs, then No new records are created and the existing user receives a JWT session
- Given The IdP ID token has an exp 3 hours from now, when The JWT session is issued, then The session expiry is set to 3 hours, not 24 hours
- Given An SSO callback completes with an email that exists as a password account with no SSO identity, when The conflict check runs, then No session is issued and the account-linking flow is triggered instead (per EP-3)

**Blocked by:** `SS-2`, `SS-5`, `SS-7`

_Estimate rationale: Core business logic touching user creation, identity linking, and session issuance with multiple branch conditions; 2.5 days._
