## Summary

A guest checkout path is being added to an existing e-commerce platform so shoppers can complete purchases without creating an account, reducing cart abandonment by ~20%. The flow covers checkout entry, address validation, Stripe Elements payment (including Apple Pay/Google Pay), order creation, and transactional email. The implementation must be PCI DSS compliant, mobile-friendly, and handle edge cases such as cart expiry, payment retries, and idempotent order creation.

## Assumptions

- An email delivery service (e.g., SendGrid, Postmark, or SES) is already provisioned or can be quickly configured — no procurement delay is assumed.
- The existing cart system stores cart state server-side and can be referenced by the guest session; no cart refactor is required.
- A postal/address validation API provider (e.g., SmartyStreets or USPS Web Tools) is approved and accessible; API keys can be obtained within the sprint.
- Stripe account is already set up and test mode is available; the team has access to Stripe Dashboard to configure webhooks and API keys.
- The existing analytics client (e.g., Segment, Amplitude) is already instrumented on the frontend and supports custom event calls.
- US-only shipping means no currency or international address complexity in v1.
- The existing 'Sign In' flow at the checkout entry point does not need modification, only the redirect from cart to checkout changes.
- CI/CD pipeline and a staging environment exist and can run E2E tests against Stripe test mode.
- No A/B testing framework is required at launch — the new flow simply replaces the forced-registration redirect for all users.

## Risks

- Stripe Elements LCP impact: loading the Stripe.js script asynchronously is required to meet the 2-second LCP target; if Stripe's CDN is slow, this may be hard to mitigate without significant architectural changes.
- Postal API availability: if the chosen postal validation provider has downtime or rate limits, address validation failures could block orders; the degraded-mode fallback (SS-5) must be implemented and tested before launch.
- Email cross-client rendering: HTML email rendering in Outlook is notoriously inconsistent; without dedicated email testing tooling (e.g., Litmus or Email on Acid), the 'renders correctly in Outlook' acceptance criterion may require significant iteration time not fully accounted for.
- Idempotency under race conditions: if two concurrent requests arrive with the same idempotency key before the first completes, a naive implementation may double-charge; the backend must use a database-level unique constraint or distributed lock, adding implementation risk.
- Cart session storage approach: if the existing cart is client-side only (e.g., localStorage), a server-side guest session referencing cart contents requires a non-trivial migration of cart data at session creation time.
- Mobile wallet availability testing: Apple Pay requires an HTTPS environment with a registered merchant domain; staging environments may not satisfy this requirement, making Apple Pay E2E testing difficult without a production-like setup.
- Sprint capacity: the total estimated scope is approximately 76 story points across all tickets, which exceeds one sprint's velocity (44 points). The team must prioritise core payment and order creation epics in Sprint 1 and defer Apple Pay, shipping notification, performance optimisation, and full E2E suite to Sprint 2.

#### `SS-1` — Data model: guest session and guest order schema

**3 pts** · backend · high · Sprint 1

> As a backend engineer, I want a well-defined schema for guest sessions and guest orders so that all downstream services have a consistent data contract.

Define and migrate database tables/collections for guest sessions (session token, cart snapshot, expiry timestamp) and guest orders (email, shipping address, Stripe payment method ID, last-4, expiry, order status). No account FK required. Schema must not store full card numbers.

**Acceptance criteria:**

- Given the migration is applied to a clean database, when a guest session record is inserted, then it persists with session token, cart JSON, created_at, and expires_at fields
- Given a guest order record is created, when its fields are inspected, then it contains email, shipping address, stripe_payment_method_id, card_last4, card_expiry, and order status — and no raw PAN field exists
- Given the schema is reviewed, when a foreign key to the users table is checked, then no such constraint exists, confirming account-free operation

_Estimate rationale: Schema design plus migration and peer review is a solid 1.5-day effort that unblocks all other backend work._


#### `SS-2` — Backend: guest session create/read/expire API

**3 pts** · backend · high · Sprint 1

> As the checkout frontend, I want a guest session API so that I can create, retrieve, and detect expiry of a guest cart within 30 minutes of inactivity.

Implement POST /sessions/guest (creates session, returns token), GET /sessions/guest/:token (returns session or 410 if expired), and a background job or TTL mechanism that marks sessions inactive after 30 minutes. Session token must be a cryptographically random value.

**Acceptance criteria:**

- Given a POST request is made to /sessions/guest, when the response is received, then it returns HTTP 201 with a session token and expires_at timestamp 30 minutes in the future
- Given a valid session token exists, when GET /sessions/guest/:token is called within 30 minutes, then it returns the session data with HTTP 200
- Given a session has been inactive for more than 30 minutes, when GET /sessions/guest/:token is called, then the API returns HTTP 410 Gone

**Blocked by:** `SS-1`

_Estimate rationale: Three endpoints plus expiry logic and token generation is roughly 1.5 days of focused backend work._

#### `SS-7` — Backend: Stripe payment intent creation endpoint

**3 pts** · backend · high · Sprint 1

> As the checkout frontend, I want a backend endpoint that creates a Stripe PaymentIntent so that the client can securely collect payment details without raw card data touching our servers.

Implement POST /payments/intent that accepts the order amount and currency, creates a Stripe PaymentIntent server-side, and returns the client_secret. The endpoint must be authenticated to the active guest session. Amount must be calculated server-side from the cart, not passed by the client, to prevent tampering.

**Acceptance criteria:**

- Given a valid guest session token and cart are present, when POST /payments/intent is called, then a Stripe PaymentIntent is created and the client_secret is returned in the response
- Given the client attempts to pass a custom amount in the request body, when the endpoint processes the request, then the server ignores the client-supplied amount and uses the server-calculated cart total
- Given no valid guest session token is provided, when POST /payments/intent is called, then the endpoint returns HTTP 401

**Blocked by:** `SS-1`, `SS-2`

_Estimate rationale: Stripe SDK integration, server-side amount calculation, and session auth is 1.5 days of backend work._


#### `SS-8` — Backend: Stripe webhook handler and payment event audit log

**5 pts** · backend · high · Sprint 1

> As the platform, I want to handle Stripe webhook events so that payment status is kept in sync and all payment events are durably logged for audit without storing PAN data.

Register and verify a Stripe webhook endpoint (POST /webhooks/stripe). Handle payment_intent.succeeded, payment_intent.payment_failed, and charge.refunded events. Write every event to an audit log table with event type, Stripe event ID, payment intent ID, amount, timestamp — explicitly excluding any card number fields.

**Acceptance criteria:**

#### `SS-9` — Frontend: Stripe Elements payment form with retry logic

- Given a shopper successfully completes payment, when the PaymentIntent is confirmed, then the frontend proceeds to order creation and then to the confirmation page

**Blocked by:** `SS-7`, `SS-3`

_Estimate rationale: Stripe Elements setup, PaymentIntent confirmation flow, retry state machine, and error UX across devices is a full 4-day effort._

#### `SS-11` — Backend: idempotent guest order creation endpoint

**8 pts** · backend · high · Sprint 2

> As the checkout system, I want an idempotent order creation endpoint so that double-tapping 'Place Order' or network retries never result in duplicate charges or duplicate orders.

Implement POST /orders/guest. Accept an idempotency key (client-generated UUID) in the request header. On first call: capture the Stripe PaymentIntent, create the order record, and return 201. On duplicate call with the same key: return the original 201 response from cache. Store only last-4, card expiry, and Stripe payment method ID — never the full PAN.

**Acceptance criteria:**

- Given a valid guest session, confirmed PaymentIntent, and idempotency key are provided, when POST /orders/guest is called, then the payment is captured, an order is created with a unique order number, and HTTP 201 is returned
- Given the same idempotency key is used in a second request, when POST /orders/guest is called again, then HTTP 201 is returned with the same order number and no second charge is made to Stripe
- Given the PaymentIntent capture fails, when POST /orders/guest is called, then the order is not created and HTTP 402 is returned with a descriptive error

**Blocked by:** `SS-1`, `SS-7`, `SS-8`

_Estimate rationale: Idempotency key storage, Stripe capture, transactional order write, and failure rollback logic is a careful 4-day backend task._
